The Problem

Most teams building with LLMs test their prompts manually. They try a few examples, it looks good, and they ship. The problem is that manual testing misses edge cases - and in support ticket routing, edge cases mean real customers get misrouted or ignored.

This project demonstrates eval-driven development: define what correct looks like, measure it automatically on every version, and let the data tell you when something breaks or regresses.

Failure Analysis

Failure 1 - Polite complaints misclassified as feature requests

"Hi, loving the product! One small thing - the button color is a bit hard to see for me."
v1 answered
feature_request
v2 answered
complaint
The polite tone and the phrase "one small thing" caused v1 to treat this as a feature request. Fix: added explicit rule - politely worded dissatisfaction is complaint, not feature_request.

Failure 2 - Churn signals buried inside bug reports

"I want to cancel but also maybe not - depends on if you can fix the export bug I reported last month."
v1 answered
technical
v2 answered
complaint
v1 focused on the word "bug" and missed the cancellation threat. Fix: added rule - if customer mentions cancelling, always classify as complaint regardless of other content.

Key Insight - Improvements cause regressions

v2 fixed both complaint failures but introduced 2 new misclassifications on tickets v1 handled correctly. This is a real-world dynamic in LLM development. Without an eval framework running on every change, these regressions would reach production undetected. This is exactly what systematic evals are designed to catch.
