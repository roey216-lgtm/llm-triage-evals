LLM Support Triage Evaluator

Hi, I'm Roey. I'm working on building evaluation systems for LLMs because I think understanding what your model actually does is more important than just getting it to sound good. I love data analysis and figuring out where things break.

The Problem

Most teams building with LLMs test their prompts manually. Try a few examples, it looks good, ship it. The problem is that manual testing misses edge cases. In support ticket routing, edge cases mean real customers getting routed wrong or ignored.

This project shows what happens when you actually measure instead of guess. I built 150 test cases, ran three different prompt versions against them, and tracked what fails and why.

The surprising part: v2 fixed some cases but broke others. Overall, it's identical to v1. This taught me that improving prompts isn't linear. Adding more rules can help one case but hurt another. You only know if something actually improved if you measure it.

What Actually Fails

Failure Pattern 1: "I love it but..." gets misclassified as feature request

"I love your product but the UI redesign last month made things harder to find. Can we have a classic mode?"
Expected: Complaint
The core message is dissatisfaction
Model said: Feature Request
Focused on the feature ask, not the criticism
The model saw "Can we have a classic mode?" and classified it as a feature request. But the real issue is that the redesign made the product worse. The positive opening ("I love your product") masks the complaint. The classifier needs to recognize that polite dissatisfaction is still dissatisfaction.

Failure Pattern 2: Urgency gets overestimated on billing issues

"Invoice discrepancy: line item shows incorrect quantity."
Expected: Urgency = Medium
Manual accounting correction, not service breaking
Model said: Urgency = High
Words like "discrepancy" and "incorrect" trigger urgency
The model sees "incorrect" and treats it as critical. But in billing, an invoice discrepancy is annoying, not emergency level. It needs manual correction but the service is working. The opposite failure also happens: casual language about real issues gets downgraded to low urgency.

Failure Pattern 2b: Competitive pressure gets ignored

"It would be really cool if you added a Slack integration. Many of our competitors have it."
Expected: Urgency = Medium
Competitive pressure is a signal
Model said: Urgency = Low
Polite language reduces urgency
"Many of our competitors have it" is a competitive threat signal. But the casual tone and polite framing ("would be really cool") made the model downgrade this to low priority. Politeness shouldn't erase the actual business signal.

Failure Pattern 3: Support quality complaints get classified as feature requests

"Your support is helpful but slow. We're paying for premium and still wait 24 hours for responses."
Expected: Complaint
Paying customer expressing dissatisfaction
Model said: Feature Request (sometimes)
Interpreted the critique as a request for faster support
This is tricky. The ticket mentions both: "support is helpful" (positive) plus "wait 24 hours" (negative). It could be read as either a complaint or a feature request for faster response times. The classifier needs to understand context. A paying premium customer expressing frustration is a complaint, even if improvement could also be framed as a feature.

What I Learned

1. Evaluation revealed the real problems faster than intuition. I thought I understood the failure modes by hand testing a few examples. The 150 case evaluation showed completely different patterns. Edge cases matter.
2. Improving prompts has diminishing returns very quickly. Going from zero shot to guided examples helped (69.33% to 70.67%). But adding more rules and complex instructions didn't help. The ceiling seems to be around 70 71% with this dataset and model. Beyond that, you probably need better data or a better model.
3. Your evaluation methodology matters as much as your prompt. We found out halfway through that the evaluator was failing valid JSON wrapped in markdown. Fixed that, and results jumped 20 points. A bad evaluation tells you nothing useful.
4. Real failures are messier than you expect. Most failures aren't "model was dumb." They're things like: polite language obscuring urgency, or the boundary between complaint and feature request being genuinely ambiguous. That's the interesting stuff.

What's Next

If I kept working on this, I'd focus on:

  • Test against Claude Sonnet and GPT 4 to see if Haiku's limitations are hardware specific
  • Manually review the 30% failure cases and see if labeling was wrong or if the task is genuinely hard
  • Add confidence scoring to see if the model knows when it's uncertain
  • Build a confusion matrix to understand which category pairs get confused most often
  • Introduce historical ticket context to see if the model improves with more information
